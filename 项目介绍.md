# Word 论文自动排版项目

## 简介

从模版 docx/pdf + 文本资料 docx（或 Markdown）→ 一键生成符合格式的排版 docx。小白只需把文件放入 `Templates/` 和 `Inputs/`，让 Agent 运行 `python run_pipeline.py --agent-auto`，系统会自动扫描识别；若中断，会明确告诉下一步。

核心思路：**格式与内容分离**。模版提供字体/字号/行距/边距等格式参数，文本资料提供章节/段落/图片/参考文献等内容。流水线自动提取两者，生成可直接运行的 python-docx 构建脚本，最终输出排好版的 docx。

## 流水线六阶段

```
Templates/模版.docx ──→ [Phase 1] format_extractor.py ──→ Outputs/format.json
    或 .md (# 格式说明)      或 md_parser                  Outputs/格式提取.md

format.json ─────────→ [Phase 2] template_profiler.py ─→ Outputs/template_profile.json

Inputs/内容.docx/.md ──→ [Phase 3] content_parser.py ──→ Outputs/content.json
                         或 md_parser                     Outputs/内容提取.md

format.json ──┬──→ [Phase 4] script_generator.py ──→ build_generated.py
content.json ─┘                                       (所有格式值来自传参，零硬编码)

build_generated.py ──→ [Phase 5] python 运行 ──→ 最终论文.docx
                    └──────────────→ build_manifest.json
最终论文.docx ──→ [Phase 6] qa_checker.py ──→ qa_report.json / qa_report.md
                    可选 qa_visual.py ─────→ visual_report.json / visual_report.md
```

每个阶段有内置验证：格式提取独立运行两次并在不一致时第三轮仲裁；内容提取使用隔离运行目录，先比对章节结构、图片和参考文献摘要，无法形成多数一致但章节结构稳定时，最多 5 轮增量合并可恢复的图片、公式、表格和参考文献，仍无法收敛才抛出 `VerificationError`。

## 格式提取（Phase 1）

### 双重来源，文字说明优先

1. **文字说明提取**（`_parse_text_instructions`）：解析模板中嵌入的格式说明文字
   - 如 `（一级标题小三号，加粗，居中；英文Times New Roman体）` → H1 = 15pt CENTER bold
   - 如 `（正文1.5倍行距，段落首行缩进2字符，两端对齐书写）` → body LS=1.5 indent=0.74cm
   - 自动区分 TOC 格式说明和正文格式说明（取包含"加粗"的最长匹配）
   - 中文号数精确映射：小三=15, 四号=14, 小四=12, 五号=10.5

2. **OOXML 直读**（`StyleResolver`）：直接解析 `w:rPr/w:sz`、`w:rFonts`、`w:jc` 等 XML 元素
   - 解决 python-docx API 的 `None` 问题（API 在样式继承时返回 None）
   - 通过 `styles.xml` 样式树向上追溯 base style，补齐继承值
   - 最终所有 run 的 font/size/bold/italic 全部非 None（0% None 率）

### 格式参数

提取的格式参数全部存入 `P{}` 字典，包括：
- 页面尺寸、四边距
- 正文字体/字号/行距/对齐/首行缩进
- 三级标题的字号/字体/加粗/对齐
- 页眉/页脚格式
- 封面结构（标题文字、表格标签、字体字号）
- 参考文献格式（[N] 编号、悬挂缩进）

## 内容提取（Phase 2）

### 标题层级检测

- 直接读 OOXML `w:rPr` 获取字号和粗体（绕过 python-docx API 的 None 问题）
- 排除封面日期/格式备注/公式/作者单位等非标题模式
- 支持长段落标题识别（如 `Abstract: body text` → heading="Abstract:" + body 分离）

### 图片提取

- 从段落和表格单元格 relationships 中提取内嵌图片，保存到当前输出目录的 `figures/`
- 使用纯 ASCII 安全文件名
- 页眉/页脚图片记录到 `content._meta.non_body_images`，由 QA 提示为不支持迁移的非正文图片

## 脚本生成（Phase 3）

### 核心原则：零硬编码，全部传参

每个格式值要么来自 `format.json`（`P[...]`），要么是衍生计算（`D[...]`）：

| 参数 | 来源 |
|------|------|
| 页面尺寸/边距 | `P['page_w/h/mt/mb/ml/mr']` |
| 正文字体/字号 | `P['body_font/size']` (文字说明优先) |
| 行距 | `P['body_ls']` (文字说明优先) |
| 首行缩进 | `P['body_indent']` (文字说明优先) |
| 三级标题 | `P['h_levels']` (文字说明优先，OOXML 补充) |
| 页眉 | `P['header']` (OOXML 提取，剔除格式备注) |
| 封面文字/标签/字体 | `P['_text_rules']['cover']` (表格提取) |
| 封面空行数 | 从模版 body 逐元素计数 |
| 页脚字号 | `D['footer_size']` = body_size × 0.85 |
| 参考文献字号 | `D['ref_size']` = body_size - 2 |
| 正文图片宽度 | 按正文文本宽度适配 |
| 封面图片尺寸 | 从模板 `wp:extent` 读取宽高并锁定 |
| 表格单元格字号 | `D['table_cell_size']` = body_size - 3 |

### 分页算法

```
text_w_pt   = (page_w - ml - mr) / 2.54 × 72
avg_char_w  = body_size × font_ratio  (TNR=0.42, 宋体=1.0)
cpl         = text_w_pt / avg_char_w           ← 精确每行字符数
line_h      = body_size × body_ls              ← 精确行高
usable_cm   = page_h - mt - mb                 ← 精确理论可用高度，无 fudge

_ph(): 段落高度 = round(chars/cpl) × line_h     ← 四舍五入，不累积 ceil 误差
_th(): 表格高度 = body_size×2.5 + (n-1)×body_size×1.8
image: 从 wp:extent 读 EMU → pt               ← 精确图片高度
```

### 生成的脚本包含

- 封面（标签/字体/空行全从模版提取）
- 标题 + 摘要 + 关键词
- 正文（含层级标题、图片 + Fig 图注）
- 三线表（OOXML 直写，顶粗线/表头细线/底粗线）
- 交叉引用（w:anchor 书签，正文 [N] 蓝色上标 → 参考文献跳转）
- 参考文献（Acta [N] 格式，悬挂缩进，自动去重 [N] 前缀）
- 文本公式识别与原生 OOXML Math 插入（`m:oMathPara`，WPS/Word 可编辑）
- QA 检测报告（检查关键输出、样式 profile、目录、图片、公式和 LaTeX 错误，并按模式标注修复目标）
- 页眉（从模版提取文字和格式，剔除中文备注）
- 页脚（PAGE 域代码，动态页码）
- A4 自动分页

## 可能存在的问题

1. **字符宽度估算**：`font_ratio` 是标准度量值，但实际渲染受系统字体、WPS/Word 版本影响。极端情况下分页位置可能偏差 1-2 行。

2. **混合中英文段落**：中英混排段落的每行字符数难以精确。当前按 body_font 查表，如果段落内两种文字占比相近，估算会偏。

3. **封面内容填写**：优先从内容文档识别题目、姓名、学号、学院、班级、导师等字段；识别不到的字段仍需用户补齐或在内容源中规范提供。

4. **公式和特殊符号**：源 docx 没有原生 `m:oMath` 时，流水线会按文本公式重建为 OOXML Math；`_ph()` 对包含数学符号、上下标的段落仍可能低估行数。复杂多行对齐公式应优先使用真正的 OMML 结构，避免用矩阵伪装对齐导致 WPS 渲染分隔线。

5. **字体回退**：如果系统未安装楷体_GB2312 或 宋体，WPS/Word 会退回默认字体。`w:eastAsia` 已设置但无法强制嵌入字体文件。

6. **Office Viewer 预览**：VSCode 的 Office Viewer 插件不渲染页眉页脚，页码在插件中不可见，需用 Word/WPS 打开查看。

7. **复杂表格**：合并单元格、嵌套表格的高度估算为近似值。

8. **图片路径**：内容 docx 中的图片提取到本次输出目录 `figures/`，不同运行互不覆盖。表格单元格图片目前会被保留下来并进入正文图片流，但不会完全保留在原表格单元格位置；如果需要位置 100% 保真，应继续增强表格内图片渲染。

## 操作指南

1. 把模版 docx 放入 `Templates/`
2. 把文本资料 docx 或 `.md` 放入 `Inputs/`
3. 普通用户优先让 Agent 运行 `python run_pipeline.py --agent-auto`
   或运行 `python run_pipeline.py`（交互模式会询问普通用户/开发者）
   或 `python run_pipeline.py --mode user --template X --content Y`（普通用户参数模式）
   或 `python run_pipeline.py --mode developer --template X --content Y`（开发者参数模式）
   或 `python run_pipeline.py --mode user --md 论文.md`（纯 MD 模式，格式+内容合一）
4. 查看 `Outputs/<日期_文件名>/qa_report.md`、`格式提取.md` 和 `内容提取.md` 核对
5. 需要产品级检查时运行 `python run_pipeline.py --qa-level visual --mode developer --template X --content Y`
6. 普通用户微调：让 AI 修改 `Outputs/<日期_文件名>/build_generated.py`，再运行该脚本生成当前最终稿
7. 开发者沉淀通用修复：修改 `Paper_Project/Program/pipeline/` 下的核心脚本，运行 `python Paper_Project/Program/pipeline/regression_suite.py`，再重跑流水线
8. 最终 docx 在 `Outputs/<日期_文件名>/最终论文.docx`
