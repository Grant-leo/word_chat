# python-docx 基础操作速查

## 一键工作流：run_pipeline.py

```
小白操作步骤:

 1. 把模版 docx/pdf 放入 Templates/ 文件夹
 2. 把文本资料 docx 或 .md 放入 Inputs/ 文件夹
 3. 让 Agent 先检查基础依赖：
    python --version
    python -c "import docx, lxml; from PIL import Image; print('OK')"
    缺失时安装：python -m pip install python-docx Pillow lxml
 4. 普通用户不需要自己打开终端。默认让 Agent 使用自动入口：
    python run_pipeline.py --agent-auto
    该入口会自动扫描 Templates/ 和 Inputs/，单候选直接运行，多候选时只问用户选择文件名，并在预检报告里列出“可以直接回复：使用 Templates/... 作为模板 / 使用 Inputs/... 作为内容”的候选句；预检报告还会固定列出“文件应该放哪里”：模板放 Templates/（.docx/.pdf），内容放 Inputs/（.docx/.md）；
    默认普通用户模式、开启自动修复闭环，并生成 agent_summary.md/json。
    如果交互选择被取消或输入流中断，下一步默认改用该自动入口，或显式指定 --template/--content。
 5. 开发者或高级复现时才使用显式命令：
    交互模式：python run_pipeline.py（自动扫描文件，编号选择）
    或普通用户模式：python run_pipeline.py --mode user --template 模版.docx --content 内容.docx
    或开发者模式：python run_pipeline.py --mode developer --template 模版.docx --content 内容.docx
    或产品级检查：python run_pipeline.py --mode developer --qa-level visual --template 模版.docx --content 内容.docx
    或普通用户自动修复闭环：python run_pipeline.py --mode user --auto-repair --template 模版.docx --content 内容.docx
    开发者回归测试：python Paper_Project/Program/pipeline/regression_suite.py
    纯 MD 模式：python run_pipeline.py --mode user --md 论文.md（格式+内容合一）
 6. 得到 Outputs/{日期_文件名}/ 目录：
    - agent_summary.md/json → 给用户和 Agent 的第一摘要：最终文件、QA 状态、自动修复、结构/strict/visual QA 问题码下一步和人工检查项
    - 格式提取.md       → 核对模版格式是否完整
    - 内容提取.md       → 核对内容是否完整
    - template_profile.md/json → 模板能力画像和风险标记
    - workflow_mode.json → 本轮用户/开发者模式
    - qa_report.md/json  → 自动 QA 检测报告、首个问题码下一步、修复目标和 `先打开这些文件` 核对入口；生成脚本构建失败时也会写出 `MISSING_DOCX` 报告；结构 QA 自身不可用或运行崩溃时写出 `STRUCTURAL_QA_UNAVAILABLE` / `STRUCTURAL_QA_FAILED` 报告
    - repair_loop_report.md/json → 自动修复闭环审计报告（--auto-repair），顶部含下一步、修复范围和恢复命令
    - conformance_report.md/json → strict DOCX/XML 合规 QA；Markdown 报告的 `核对入口` 会列出最终 DOCX、内容摘要、content.json、build_manifest.json、template_requirements.json 和 format.json
    - visual_report.md/json → PDF 渲染 QA（--qa-level visual）；Markdown 报告的 `诊断产物` 会列出 Word/WPS PDF、文本诊断和样张 PNG 路径
    - visual_qa/        → 本地 PDF 和抽样 PNG（不提交）
    - figures/          → 本次内容文档提取的图片
    - assets/           → 本次模板提取的封面/Logo 资源
    - build_generated.py → 自动生成的构建脚本
    - 最终论文.docx     → 最终 docx

  7. 普通用户自动修复：优先让 Agent 使用 `--agent-auto` 或 `--auto-repair`，自动读取 QA、只改 build_generated.py、重建并重跑 QA
  8. 普通用户微调：让 AI 修改 Outputs/{目录}/build_generated.py
  9. 微调后由 Agent 运行：python Outputs/{目录}/build_generated.py
 10. 开发者沉淀通用修复时，再修改 Paper_Project/Program/pipeline/ 下的核心引擎脚本并重跑流水线
```

依赖说明：
- 普通流水线和 strict QA：Python 3.10+、`python-docx`、`Pillow`、`lxml`。
- 自动目录页码：可选 `pywin32` + Microsoft Word COM；不可用时保留静态可见目录行。
- visual QA：Windows PowerShell + Microsoft Word COM 导出 PDF，Poppler 命令行工具 `pdfinfo`、`pdftotext`、`pdftoppm` 做页数、文本和 PNG 抽样检查。
- WPS 交叉渲染：可选 WPS COM；除非使用 `--require-wps`，否则 WPS 不可用只作为 warning。

### 流水线架构

```
Templates/模版.docx  ──→ [Phase 1] format_extractor.py  ──→ Outputs/格式提取.md
    或 .md (# 格式说明)      或 md_parser                    Outputs/format.json

format.json ─────────→ [Phase 2] template_profiler.py ──→ Outputs/template_profile.json

Inputs/内容.docx/.md  ──→ [Phase 3] content_parser.py    ──→ Outputs/内容提取.md
                         或 md_parser                       Outputs/content.json

format.json ──┬──→ [Phase 4] script_generator.py ──→ build_generated.py
content.json ─┘

build_generated.py ──→ [Phase 5] python 运行 ──→ 最终论文.docx
最终论文.docx ──→ [Phase 6] qa_checker.py ──→ qa_report.json / qa_report.md
                    可选 qa_visual.py ─────→ visual_report.json / visual_report.md
```

当前规则：
- 每次运行生成独立输出目录；同一天同名内容会自动追加 `_2`、`_3` 后缀。
- 输入论文、模板、输出目录、生成 docx/PDF/PNG 都属于本地隐私数据，不应提交到 Git；`Inputs/`、`Outputs/`、`Templates/` 只保留占位 txt；Codex 本地 skill 是用户运行环境配置，不放进仓库或 `docs/skills/`。
- `--mode user` 只修当前输出目录的 `build_generated.py`；`--mode developer` 只修 `pipeline/` 核心引擎。
- `--auto-repair` 是受控自动修复闭环：只允许修改当前输出目录的 `build_generated.py`，每轮重建 DOCX 并重跑已启用的结构/strict/visual QA，写入 `repair_loop_report.md/json`；连续无改善、重建失败、达到轮次上限或遇到缺图/扫描 PDF/内容缺失等需要用户补文件的问题时停止。strict/visual 依赖缺失时必须失败并报告，不能假装收敛；自动修复内部的结构/strict/visual QA 如果不可用或运行崩溃，也必须写出对应 QA 报告和 `repair_loop_report.md/json`，使用 `STRUCTURAL_QA_UNAVAILABLE`、`STRUCTURAL_QA_FAILED`、`CONFORMANCE_QA_FAILED` 或 `VISUAL_QA_FAILED`，脱敏异常细节，并在能推断 workflow 时保留完整重跑命令。停止时报告必须给出 `next_action`、`resume_scope` 和 `resume_command`，并由 `agent_summary.md/json` 汇总。
- 任意预检、QA、依赖、构建或自动修复中断，都必须给普通用户明确下一步；预检阶段写 `Outputs/_agent_preflight_latest/agent_preflight_report.md/json`，多候选时把每个候选文件转成可直接回复给 Agent 的句子，并在 Markdown `文件应该放哪里` 与 JSON `source_folders` 中列出 `Templates/`（`.docx`/`.pdf`）和 `Inputs/`（`.docx`/`.md`）的放置规则；正式输出阶段写/读 `agent_summary.md/json` 和对应 QA 报告，不能让用户干等。若 `build_generated.py` 执行失败导致最终 DOCX 未生成，也必须写出 `qa_report.md/json`、`qa_repair_plan.md/json` 和 `qa_fix_prompt.txt`，用 `MISSING_DOCX` 指向本次 `build_generated.py` 的修复和重建命令。若 `qa_checker.py` / `qa_checker_modules` 自身不可用，流水线也必须写出 `qa_report.md/json`、`qa_repair_plan.md/json` 和 `qa_fix_prompt.txt`，使用 `STRUCTURAL_QA_UNAVAILABLE`，脱敏本地路径，`resume_scope=full_pipeline`，能从 `workflow_mode.json` 推断时保留完整重跑命令，并明确下一步是修复结构 QA 依赖后重跑完整流水线。若结构 QA 开始运行后崩溃，写出 `STRUCTURAL_QA_FAILED` 并走同一套结构 QA 修复向导；若 strict 或 visual QA 开始运行后崩溃，分别写出 `CONFORMANCE_QA_FAILED` 或 `VISUAL_QA_FAILED` 到对应报告，脱敏异常细节，并明确修复对应 QA 模块/渲染依赖后重跑。结构/strict/visual QA 失败时，`agent_summary.md/json` 必须直接聚合前几个问题码和“小白用户下一步”；结构 QA 只有 warning 时也不能简单写“已通过”就结束，必须在 `qa_report.md/json`、`qa_repair_plan.md/json` 和 `agent_summary.md/json` 中给出 warning 问题码、用户动作和是否需要重跑；`qa_repair_plan.md/json` 对 warning-only 结构 QA 应使用 `resume_scope=warning_review`，摘要写清“没有阻断错误但有警告”，并在能推断 workflow 时保留重跑命令；strict/visual QA 只有 warning 时也不能简单写“已通过”就结束，必须在 `conformance_report.md/json`、`visual_report.md/json` 和 `agent_summary.md/json` 中给出 warning 问题码、人工确认点和恢复/重跑动作；所有 warning-only Markdown 报告首页结果行必须写“通过但有警告”或“已通过但有警告”，不能只写普通“通过”；运行时终端 QA 摘要也必须写“通过但有警告”、展示 warning 数，并打印该报告的 `next_action`，避免用户只看到 `[OK]` 后误以为完全无需处理；结构 `qa_report.json`、strict `conformance_report.json`、visual `visual_report.json` 以及依赖缺失、QA 崩溃、构建失败、提取验证失败等 fallback 报告都必须显式写出 `status`（`passed` / `passed_with_warnings` / `failed`）和 `result_label`，契约检查必须覆盖结构/strict/visual 三类报告并报告缺失或与 `passed`/warning 状态不一致的字段；`agent_summary.md/json` 中每个结构/strict/visual QA 单项也要同步保留 `status` 与 `result_label`，单项通过但有 warning 时同样写“通过但有警告”；结构 QA 的 `qa_report.md/json` 顶部 `next_action` 也必须点名首个问题码和具体动作；`qa_repair_plan.json` 顶部必须写出 `next_action`、`resume_scope`、`resume_command`，明确是补输入文件、重建当前 DOCX、重跑完整流水线、复核 warning，还是做最终 Word/WPS 核对；strict/visual 报告遇到占位符、Word 域、strict 样式/页边距/内容 warning、PDF 页数无效、页面 PNG 不可读、渲染依赖缺失、缺黄金基线、空白页/目录 warning、可选 WPS 导出 warning、`WPS_PDFINFO_UNAVAILABLE`、`WPS_PDFINFO_FAILED`、`WPS_PAGE_COUNT_INVALID`、`WPS_PAGE_COUNT_MISMATCH`、`WPS_PAGE_SIZE_MISMATCH`、`WPS_TEXT_PAGE_MISMATCH`、`WPS_SAMPLE_RENDER_FAILED` 或 `WPS_SAMPLE_IMAGE_MISMATCH` 时，也要在顶部下一步写清楚具体处理方式；其中 WPS PDF 信息不可读或页数无效必须告诉用户检查 WPS PDF/DOCX、修复 PDF/Poppler 或空白输出原因并重跑 visual QA，WPS/Word 页数不一致必须告诉用户先比对分页差异，WPS/Word 页面尺寸或横竖方向不一致必须告诉用户比对纸张大小和页面设置，WPS/Word 可提取文本页数明显不一致必须告诉用户比对正文、目录、公式和图片内容，并用 `visual_qa/rendered_word.txt` 与 `visual_qa/rendered_wps.txt` 辅助定位；WPS 样张渲染失败或样张画面明显不一致必须告诉用户比对 `visual_qa/samples/` 和 `visual_qa/wps/samples/`，重点检查公式、图片、表格和正文画面，修复后重跑 visual QA。
- `workflow_mode.json` 只能为本项目 `Inputs/` / `Templates/` 下可安全表示的文件生成可复制重跑命令；如果高级用户传入了这些目录外的绝对路径，即使外部路径里也有同名 `Inputs` 或 `Templates` 目录，不要把它折叠成只有文件名的假命令，报告应提示先把文件放入本项目 `Inputs/` 或 `Templates/`，再按文件名重跑。
- 交互式文件/模式选择被用户取消或 stdin 中断时，也必须提示下一步：普通用户优先运行 `python run_pipeline.py --agent-auto`，已知文件名时用 `--template` / `--content` 重跑。
- 独立调试入口的派生产物默认写入 `Outputs/_...`：`format_extractor.py` → `Outputs/_format_extractor_cli/`，`content_parser.py` → `Outputs/_content_parser_cli/`，`md_parser.py` → `Outputs/_md_parser_cli/`；默认 API 运行目录也应在 `Outputs/_format_extractor_extract/`、`Outputs/_content_parser_extract/`、`Outputs/_md_parser_extract/`，不能污染 `Inputs/` 或 `Templates/`。
- `qa_checker.py` 只输出检测报告和修复目标，不自动改代码；最终仍要用 WPS/Word 做视觉核对。
- `template_profiler.py` 生成模板画像；后续通用规则应优先依赖画像能力/风险标记，不写学校名、文件名硬编码。
- `format_extractor.py` 是模板格式提取稳定入口；新的模板解析规则优先沉淀到 `format_extractor_modules/` 子模块中。目前 OOXML 数值换算、段落度量、样式继承解析、语义样式画像、封面资产和封面表格布局抽取已经子模块化。
- `--qa-level visual` 会用 Word COM 导出 PDF，并用 Poppler 工具做页数、纸张、文本和抽样 PNG 检查；抽样页数量保持有界，但会优先覆盖封面、目录/正文起点，以及能从 PDF 文本识别到的图片、表格、公式风险页，避免只检查前几页；WPS 可用时还会交叉检查 WPS/Word 页数、页面尺寸/横竖方向、可提取文本页数和同页 PNG 样张感知哈希，并分别保留 `visual_qa/rendered_word.txt`、`visual_qa/rendered_wps.txt`、`visual_qa/samples/` 与 `visual_qa/wps/samples/`；`visual_report.md` 的 `诊断产物` 会直接列出这些 PDF、文本和样张路径，便于小白用户按报告打开检查；缺少必要渲染工具时 visual QA 不算通过，并使流水线失败退出。
- strict/visual 报告顶部 `next_action` 必须点名 leading issue code，再给出中文下一步；这样用户即使只打开 `conformance_report.md` 或 `visual_report.md`，也能把 `STYLE_MISMATCH`、`PDF_PAGE_COUNT_INVALID`、显式启用 golden 时的 golden baseline 问题等问题码和具体处理动作对应起来。
- 内容图片写入当前输出目录的 `figures/`，不写回 `Inputs/`；Markdown 图片相对路径按 `.md` 文件所在目录解析，支持 `%20` 空格编码、`<带空格路径>` 包裹、文件名括号、可选图片 title（如 `![图](path "title")`）、本地 `?query` / `#fragment` 后缀、`<img src="path" alt="图">` HTML 图片标签、HTML 懒加载 `data-src` / `data-original`、`srcset` 首候选、PNG/JPG data URI，以及 `![图][id]` + `[id]: path` 引用式图片、引用定义下一行 title、`![图]` + `[图]: path` shortcut 引用式图片这类常见写法；本地 Markdown 图片只把 `.png` / `.jpg` / `.jpeg` 作为稳定可生成格式，GIF/WebP/SVG/无扩展名图片或扩展名与真实格式不一致的图片进入 `CONTENT_IMAGE_UNREADABLE` 并提示重新导出普通 PNG/JPG；data URI 也必须声明类型和真实图片格式一致，例如 `data:image/png` 里实际是 JPEG 时进入 `CONTENT_IMAGE_UNREADABLE`，不能按错误扩展名写入 `figures/`；Markdown 表格单元格中的图片要挂到 `table_cell_items` 并标记 `location="markdown_table_cell"`，生成 DOCX 时渲染在对应 Word 表格单元格内，缺失时进入同一套 QA；DOCX 正文表格单元格里的图片也要挂到源单元格 `table_cell_items` 并标记 `location="table_cell"`，同一单元格段落内图片在文字前/后的顺序要按 OOXML run 顺序保留，单元格内 inline OMML/LaTeX 公式和脚注/尾注引用要作为 `rich_text` 替换项保留并生成原生 Word 公式/注释引用，图片段落后接 LaTeX + OMML 混合公式段落时也要保持图片先于公式段落的源顺序；如果同一个源段落中图片夹在前置文字和后续 LaTeX/OMML/脚注之间，解析时应按图片边界拆成前后单元格段落，生成时保持文字、图片、公式、脚注原顺序；四层嵌套表内的单元格图片、inline 公式和注释引用要保留在嵌套表内，结构 QA 与 strict 计数必须递归统计，不能追加成表格后的普通正文图片，也不能把公式压成普通字符串或丢失注释引用；本地缺失或未定义引用图片进入 `CONTENT_IMAGE_MISSING`，本地图片文件损坏、扩展名不匹配、格式不支持或 data URI 损坏/类型不支持时进入 `CONTENT_IMAGE_UNREADABLE` 并提示重新导出普通 PNG/JPG，远程 `http://` / `https://` 图片进入 `CONTENT_IMAGE_REMOTE_UNSUPPORTED` 并提示先下载到本地、改成相对路径后重跑；代码块里的 `[id]: path` 示例不能被当作图片引用定义剥离；Windows 常见 UTF-8 BOM 开头的 YAML/front matter、Markdown H1 标题、以及 Setext `Title` + `===` 一级标题也要识别，中文题名进入 `title_cn`，英文/非 CJK 题名进入 `title_en`，避免有效标题被误报 `TITLE_MISSING`。
- `内容提取.md` 是给用户看的内容摘要：结构化图片、表格、公式要按真实角色显示，`role="image"` 的正文图/表格单元格图也要显示为 `[图片]`，表格摘要有单元格图片时要显示 `单元格图片 N`，不能落成 `[结构化内容]`；正文流中已有图片时不要再重复列章节图片清单；“图 1 展示了……”这类正文引用句不能当图注。
- 表格单元格注释边界：如果图片后只有脚注锚点、没有后续可见正文，也必须在图片后原位生成 Word 原生脚注引用；四层嵌套表单元格中同段文字、图片、LaTeX、OMML 和脚注引用也必须保持源顺序，并由递归 QA 计数覆盖。
- `content_parser.py` 是内容解析稳定入口；新的内容解析规则优先沉淀到 `content_parser_modules/` 子模块中，再由入口编排调用。目前占位符、样式、文本清洗、封面/题名、图表题注流、段落 OOXML 流、正文元素分发、源目录、图片、表格、公式抽取/修复、标题识别、参考文献收集和章节构建/后处理已经子模块化。
- `md_parser.py` 是 Markdown 格式/内容解析稳定入口；YAML/自然语言格式说明、开头格式说明块剥离、UTF-8 BOM 安全的 YAML/front matter 与 H1 识别、Setext 一级题名识别、行内/显示公式 token、Markdown/HTML 图片路径解析/缺失记录、Markdown 表格单元格图片挂载、GFM 表格解析和 Markdown 文本清洗已沉淀到 `md_parser_modules/`。
- `pipeline_runner/` 是 `run_pipeline.py` 的编排辅助包，承接 CLI/文件扫描、运行上下文、依赖加载、产物写入、提取验证、模板阶段、构建执行、JSON 契约、QA、受控自动修复闭环、终端报告和完成摘要；通用排版规则仍放在各核心引擎入口中。

- 本轮旧式 hMerge/vMerge 非矩形冲突补充：旧式横向合并和纵向合并如果没有形成同宽矩形，解析端不再输出重叠 merge，而是 fail-open 保留 continuation 文本、只保留安全横向合并；源审计把它计入异常合并网格并给出复杂表格复核提示。
- 本轮混合 gridSpan/hMerge 重复编码补充：转换工具同时写入现代 `w:gridSpan` 和旧式 `w:hMerge` 时，解析端按 `gridSpan` 只计算一次横向合并，空的重复 continuation 不再多生成一列；源审计新增 `irregular_hmerge_count`，让普通用户看到表格编码异常的复核提示。

## 三线表

所有表格统一使用三线表格式：顶粗线（1.5pt）/ 表头下细线（0.5pt）/ 底粗线（1.5pt）/ 无竖线。

需要的额外 import：

```python
from docx.oxml import OxmlElement
from docx.oxml.ns import qn
```

### 核心实现（直接复制使用）

```python
def _remove_table_borders(table):
    """移除 tblBorders，避免 insideH=none 压制单元格级边框"""
    tbl = table._tbl
    tblPr = tbl.tblPr
    if tblPr is None:
        tblPr = OxmlElement('w:tblPr')
        tbl.insert(0, tblPr)
    for old in tblPr.findall(qn('w:tblBorders')):
        tblPr.remove(old)


def _set_tc_border(cell, top=None, bottom=None, left='nil', right='nil'):
    """为单个单元格设置四边边框，不依赖继承"""
    tcPr = cell._tc.get_or_add_tcPr()
    for old in tcPr.findall(qn('w:tcBorders')):
        tcPr.remove(old)
    tcB = OxmlElement('w:tcBorders')
    tcPr.append(tcB)

    def add_border(pos, val='nil', sz='0'):
        b = OxmlElement(f'w:{pos}')
        b.set(qn('w:val'), val)
        b.set(qn('w:sz'), str(sz))
        b.set(qn('w:space'), '0')
        b.set(qn('w:color'), '000000')
        tcB.append(b)

    if top is not None:
        add_border('top', top[0], top[1])
    else:
        add_border('top', 'nil', '0')
    if bottom is not None:
        add_border('bottom', bottom[0], bottom[1])
    else:
        add_border('bottom', 'nil', '0')
    add_border('left', left, '0')
    add_border('right', right, '0')


def three_line_table(table):
    """将表格转为三线表：第0行 top=粗 bottom=细 / 第1行 top=细 / 末行 bottom=粗 / 无竖线"""
    _remove_table_borders(table)
    nrows = len(table.rows)
    if nrows == 0:
        return
    for ri, row in enumerate(table.rows):
        for cell in row.cells:
            if ri == 0 and nrows == 1:
                _set_tc_border(cell, top=('single', '12'), bottom=('single', '12'))
            elif ri == 0:
                _set_tc_border(cell, top=('single', '12'), bottom=('single', '4'))
            elif ri == 1 and ri == nrows - 1:
                _set_tc_border(cell, top=('single', '4'), bottom=('single', '12'))
            elif ri == 1:
                _set_tc_border(cell, top=('single', '4'))
            elif ri == nrows - 1:
                _set_tc_border(cell, bottom=('single', '12'))
            else:
                _set_tc_border(cell)
```

### 用法

```python

# 填表头

    set_cell(table.rows[0].cells[i], h, bold=True, size=10)

# 填数据

    for ci, text in enumerate(row_data):
        set_cell(table.rows[ri].cells[ci], text, size=9)

# ☆ 填完所有内容后再调用 ☆
three_line_table(table)
```

### 实现原理

1. `_remove_table_borders` — 完全移除 `tblPr/tblBorders`，避免 Word 的 `insideH=none` 压制单元格级边框
2. `_set_tc_border` — 为每个单元格独立写入 `tcBorders`（top/bottom/left/right），不依赖样式继承
3. `three_line_table` — 遍历全表，按行位置分配边框规则
4. 粗线 `('single', '12')` = 1.5pt，细线 `('single', '4')` = 0.5pt，无框 `'nil'`

**关键**：`three_line_table()` 必须在填完所有数据后调用，否则后续填入可能破坏边框设置。

## 交叉引用（正文 [N] ↔ 参考文献列表 可跳转）

实现学术论文中的 Ctrl+Click 跳转引用：正文蓝色上标 `[1],[2]` 点击后跳转到参考文献列表对应条目。

### 全局变量

```python
ref_bookmarks = {}  # ref_num -> bookmark_name
```

### ① 正文引用标记（B_ref + add_ref_hyperlink）

```python
def B_ref(text, ref_nums):
    """正文段落 + 末尾蓝色上标超链接引用标记，如 [1,2,3]"""
    p = doc.add_paragraph()
    J(p)
    R(p, text, size=12)
    for i, n in enumerate(ref_nums):
        if i > 0:
            R(p, ', ', size=10, superscript=True)
        add_ref_hyperlink(p, n, f'[{n}]')
    return p


def add_ref_hyperlink(paragraph, ref_num, display_text):
    """在段落中插入指向参考文献书签的锚点超链接（蓝色上标）"""
    bookmark_name = f'_Ref{ref_num}'
    ref_bookmarks[ref_num] = bookmark_name

    # w:anchor 方式 — 不需要 OPC relationship，避免 OPC 损坏
    hl = paragraph._element.makeelement(qn('w:hyperlink'), {
        qn('w:anchor'): bookmark_name,
        qn('w:history'): '1',
    })
    run_elem = paragraph._element.makeelement(qn('w:r'), {})
    rPr = paragraph._element.makeelement(qn('w:rPr'), {})

    # 上标
    v = paragraph._element.makeelement(qn('w:vertAlign'), {qn('w:val'): 'superscript'})
    rPr.append(v)
    # 字号 10pt → 20 half-pts
    sz = paragraph._element.makeelement(qn('w:sz'), {qn('w:val'): '20'})
    rPr.append(sz)
    # 蓝色
    c = paragraph._element.makeelement(qn('w:color'), {qn('w:val'): '0000FF'})
    rPr.append(c)

    run_elem.append(rPr)
    t = paragraph._element.makeelement(qn('w:t'), {})
    t.set(qn('xml:space'), 'preserve')
    t.text = display_text
    run_elem.append(t)
    hl.append(run_elem)
    paragraph._element.append(hl)
```

### ② 参考文献列表（带书签锚点）

每条参考文献必须以 `w:bookmarkStart` / `w:bookmarkEnd` 包裹，书签名与 `add_ref_hyperlink` 中的 `_Ref{N}` 一致：

```python
from docx.oxml import OxmlElement
from docx.oxml.ns import qn

refs = [
    (1, 'Weissmueller J. Alloy effects in nanostructures. Nanostructured Materials, 1993, ...'),
    (2, 'Smith C S. Grains, phases, and interfaces. Trans. Metall. Soc. AIME, 1948, ...'),
    # ...
]

for num, ref_text in refs:
    p = doc.add_paragraph()
    J(p)
    p.paragraph_format.left_indent = Cm(1.27)
    p.paragraph_format.first_line_indent = Cm(-1.27)

    # ★ 书签开始
    bookmark_name = f'_Ref{num}'
    bk_start = OxmlElement('w:bookmarkStart')
    bk_start.set(qn('w:id'), str(num))
    bk_start.set(qn('w:name'), bookmark_name)
    p._element.append(bk_start)

    R(p, f'[{num}] ', size=10)
    R(p, ref_text, size=10)

    # ★ 书签结束（id 必须与 start 一致）
    bk_end = OxmlElement('w:bookmarkEnd')
    bk_end.set(qn('w:id'), str(num))
    p._element.append(bk_end)
```

### ③ 用法

```python
# 正文中引用
B_ref('Weissmueller established the thermodynamic framework for GB energy.',
      [1])
B_ref('The Zener pinning model provides a classical kinetic stabilization mechanism.',
      [2])
B_ref('These two approaches together define the core stability landscape.',
      [1, 2])   # 多篇引用

# 参考文献列表（同②，逐条生成带书签的条目）
```

### 实现原理

| 组件 | 机制 | 目的 |
|------|------|------|
| `add_ref_hyperlink` | `w:hyperlink` 的 `w:anchor` 属性 | 跳转到同名书签，**不需要 OPC relationship**，避免 `r:id` 损坏 |
| `bookmarkStart`/`bookmarkEnd` | Word 内部书签 | 定义跳转目标，`w:name` 与 `w:anchor` 值匹配 |
| `ref_bookmarks` dict | 内存追踪 | 记录已注册的书签名，确保不重复 |
| 蓝色上标 `0000FF` + `superscript` | 视觉标记 | 与正文区分，明确可点击 |

**与外部超链接的区别**：外部 URL（`https://...`）必须用 OPC `r:id` + `RT.HYPERLINK`；内部交叉引用必须用 `w:anchor` + 书签。混用会导致 OPC 损坏或链接失效。

```python
run._element.append(fld_end)
```

### 实现原理

| 元素 | 作用 |
|------|------|
| `w:fldChar fldCharType="begin"` | 域代码开始标记 |
| `w:instrText` `PAGE` | Word 域指令，渲染为当前页码 |
| `w:fldChar fldCharType="end"` | 域代码结束标记 |
| `is_linked_to_previous = False` | 断开"链接到前一节"，多节文档必须设置 |

**为什么不用静态文本**：硬编码 `1` / `2` / `3` 不会随文档编辑后的页数变化而更新。Word `PAGE` 域是动态字段，每次打开文件或打印时自动计算正确页码。

**常见变体**：
- `{ PAGE }` — 当前页码
- `{ NUMPAGES }` — 总页数
- `Page { PAGE } of { NUMPAGES }` — "Page X of Y" 格式

## A4 自动分页

`build_acta_manuscript.py` 内置了 `paginate_a4(doc)` 函数，在 `doc.save()` 前自动调用，一个脚本即出最终 Word。

### 用法

```bash
python Program/build_acta_manuscript.py  # 排版 + 分页一键完成
```

### 算法

| 参数 | 值 |
|------|-----|
| 可用高度 | 25.0 cm ≈ 709 pt（比理论值多 ~1.5%，补偿 Word 渲染误差） |
| 每行字数（英文 12pt） | ~75 chars |
| 行高（12pt × 2.0 行距） | 24 pt |
| 每页行数 | ~29 行 |

遍历 `doc.element.body` 中所有 `w:p` 和 `w:tbl` 元素：
- 文本段：从 `w:rPr/w:sz` 取实际字号，文本长度 ÷ 75 估算行数，× 字号 × 行距得高度
- 图片段：从 `wp:extent` 取实际图片高度（EMU → pt）
- 表格：表头 28pt + 每数据行 20pt
- 累积到 ~698pt 时在当前位置前插入 `w:br type="page"`

### 防孤行（Widow/Orphan Control）

`paginate_a4()` 内置胶合（glue）逻辑，自动检测以下模式并阻止跨页拆散：

| 模式 | 检测规则 |
|------|---------|
| 短引介段 → 表格 | 文本 < 500 chars + 紧跟 `w:tbl` |
| 短引介段 → 图片 | 文本 < 500 chars + 紧跟含 `w:drawing` 的段落 |
| 图片 → 图注 | `w:drawing` 段 + 紧跟以 `Fig.`/`Table` 开头的图注 |
| 标题 → 正文 | 文本 < 100 chars + 高度 > 20pt（大字 = 标题） |

同时，关键段落设置了 Word 原生 `keep_with_next = True` 属性作为第二层保护。


## Word 脚注（OOXML 直接写入）

python-docx 不原生支持脚注。MCP safe-docx 的 `add_footnote` 有路径白名单限制（仅允许 C: 盘）。**通用方案**：直接操作 docx 的 ZIP 包内 XML 文件。

### 原理

docx 是 ZIP 包。脚注需要修改/新增 4 个内部文件：

| 操作 | 文件 | 说明 |
|------|------|------|
| 加引用标记 | `word/document.xml` | 在目标段落末尾插入 `<w:r><w:footnoteReference w:id="1"/></w:r>` |
| 建脚注内容 | `word/footnotes.xml` | 新建，含分隔符(-1/0)和实际脚注(1,2,...) |
| 注册关系 | `word/_rels/document.xml.rels` | 添加 `footnotes.xml` 的关系声明 |
| 注册类型 | `[Content_Types].xml` | 添加 footnotes 的 ContentType |

### 可直接复用的脚本

```python
import zipfile, os
from lxml import etree

def add_footnote_to_docx(docx_path, search_text, footnote_text):
    """在包含 search_text 的段落后添加脚注。直接修改 docx_path 文件。"""

    W   = 'http://schemas.openxmlformats.org/wordprocessingml/2006/main'
    R   = 'http://schemas.openxmlformats.org/officeDocument/2006/relationships'
    CT  = 'http://schemas.openxmlformats.org/package/2006/content-types'
    REL = 'http://schemas.openxmlformats.org/package/2006/relationships'
    XML_NS = 'http://www.w3.org/XML/1998/namespace'

    tmp = docx_path.replace('.docx', '_tmp.docx')

    # ── 1. 读入 ZIP ──
    with zipfile.ZipFile(docx_path, 'r') as zin:
        files = {name: zin.read(name) for name in zin.namelist()}

    # ── 2. 定位段落，插入脚注引用 ──
    doc_xml = etree.fromstring(files['word/document.xml'])
    body = doc_xml.find(f'{{{W}}}body')

    target_p = None
    footnote_id = None

    # 检查已有脚注 ID，递增
    existing_ids = set()
    for ref in doc_xml.findall(f'.//{{{W}}}footnoteReference'):
        try: existing_ids.add(int(ref.get(f'{{{W}}}id')))
        except: pass
    if 'word/footnotes.xml' in files:
        fn_xml = etree.fromstring(files['word/footnotes.xml'])
        for fn in fn_xml.findall(f'{{{W}}}footnote'):
            try: existing_ids.add(int(fn.get(f'{{{W}}}id')))
            except: pass
    footnote_id = max(existing_ids | {0}) + 1

    for p in body.findall(f'{{{W}}}p'):
        txt = ''.join(t.text or '' for t in p.findall(f'.//{{{W}}}t'))
        if search_text in txt:
            target_p = p
            break

    if target_p is None:
        raise ValueError(f'找不到包含 "{search_text}" 的段落')

    # 在段落末尾插入 <w:r><w:rPr><w:rStyle w:val="FootnoteReference"/></w:rPr><w:footnoteReference w:id="N"/></w:r>
    fn_run = etree.SubElement(target_p, f'{{{W}}}r')
    fn_rPr = etree.SubElement(fn_run, f'{{{W}}}rPr')
    fn_style = etree.SubElement(fn_rPr, f'{{{W}}}rStyle')
    fn_style.set(f'{{{W}}}val', 'FootnoteReference')
    fn_ref = etree.SubElement(fn_run, f'{{{W}}}footnoteReference')
    fn_ref.set(f'{{{W}}}id', str(footnote_id))

    files['word/document.xml'] = etree.tostring(
        doc_xml, xml_declaration=True, encoding='UTF-8', standalone=True)

    # ── 3. 创建/更新 footnotes.xml ──
    if 'word/footnotes.xml' in files:
        fn_root = etree.fromstring(files['word/footnotes.xml'])
    else:
        fn_root = etree.Element(f'{{{W}}}footnotes',
                                nsmap={'w': W, 'r': R})
        # 必须的分隔符脚注
        for fid, ftype, tag in [('-1', 'separator', 'separator'),
                                ('0', 'continuationSeparator', 'continuationSeparator')]:
            fn = etree.SubElement(fn_root, f'{{{W}}}footnote')
            fn.set(f'{{{W}}}type', ftype)
            fn.set(f'{{{W}}}id', fid)
            fp = etree.SubElement(fn, f'{{{W}}}p')
            fr = etree.SubElement(fp, f'{{{W}}}r')
            etree.SubElement(fr, f'{{{W}}}{tag}')

    # 添加实际脚注
    fn_n = etree.SubElement(fn_root, f'{{{W}}}footnote')
    fn_n.set(f'{{{W}}}id', str(footnote_id))
    fn_p = etree.SubElement(fn_n, f'{{{W}}}p')

    # 脚注内引用标记 <w:r><w:rPr><w:rStyle w:val="FootnoteReference"/></w:rPr><w:footnoteRef/></w:r>
    r1 = etree.SubElement(fn_p, f'{{{W}}}r')
    rPr1 = etree.SubElement(r1, f'{{{W}}}rPr')
    s1 = etree.SubElement(rPr1, f'{{{W}}}rStyle')
    s1.set(f'{{{W}}}val', 'FootnoteReference')
    etree.SubElement(r1, f'{{{W}}}footnoteRef')

    # 空格
    rs = etree.SubElement(fn_p, f'{{{W}}}r')
    rPrs = etree.SubElement(rs, f'{{{W}}}rPr')
    szs = etree.SubElement(rPrs, f'{{{W}}}sz'); szs.set(f'{{{W}}}val', '18')
    rfs = etree.SubElement(rPrs, f'{{{W}}}rFonts')
    rfs.set(f'{{{W}}}ascii', 'Times New Roman')
    rfs.set(f'{{{W}}}eastAsia', '宋体')
    ts = etree.SubElement(rs, f'{{{W}}}t')
    ts.set(f'{{{XML_NS}}}space', 'preserve'); ts.text = ' '

    # 脚注正文
    r2 = etree.SubElement(fn_p, f'{{{W}}}r')
    rPr2 = etree.SubElement(r2, f'{{{W}}}rPr')
    sz2 = etree.SubElement(rPr2, f'{{{W}}}sz'); sz2.set(f'{{{W}}}val', '18')
    rf2 = etree.SubElement(rPr2, f'{{{W}}}rFonts')
    rf2.set(f'{{{W}}}ascii', 'Times New Roman')
    rf2.set(f'{{{W}}}eastAsia', '宋体')
    t2 = etree.SubElement(r2, f'{{{W}}}t')
    t2.set(f'{{{XML_NS}}}space', 'preserve'); t2.text = footnote_text

    files['word/footnotes.xml'] = etree.tostring(
        fn_root, xml_declaration=True, encoding='UTF-8', standalone=True)

    # ── 4. 注册关系（仅首次） ──
    rels_path = 'word/_rels/document.xml.rels'
    rels_xml = etree.fromstring(files[rels_path])
    has_fn_rel = any('footnotes' in (rel.get('Target') or '') for rel in rels_xml)
    if not has_fn_rel:
        max_id = 0
        for rel in rels_xml:
            rid = rel.get('Id', '')
            if rid.startswith('rId'):
                try: max_id = max(max_id, int(rid[3:]))
                except: pass
        fn_rel = etree.SubElement(rels_xml, f'{{{REL}}}Relationship')
        fn_rel.set('Id', f'rId{max_id + 1}')
        fn_rel.set('Type', 'http://schemas.openxmlformats.org/officeDocument/2006/relationships/footnotes')
        fn_rel.set('Target', 'footnotes.xml')
        files[rels_path] = etree.tostring(
            rels_xml, xml_declaration=True, encoding='UTF-8', standalone=True)

    # ── 5. 注册 ContentType（仅首次） ──
    ct_xml = etree.fromstring(files['[Content_Types].xml'])
    if not any(o.get('PartName') == '/word/footnotes.xml' for o in ct_xml):
        fn_ct = etree.SubElement(ct_xml, f'{{{CT}}}Override')
        fn_ct.set('PartName', '/word/footnotes.xml')
        fn_ct.set('ContentType',
                  'application/vnd.openxmlformats-officedocument.wordprocessingml.footnotes+xml')
        files['[Content_Types].xml'] = etree.tostring(
            ct_xml, xml_declaration=True, encoding='UTF-8', standalone=True)

    # ── 6. 写回 ──
    with zipfile.ZipFile(tmp, 'w', zipfile.ZIP_DEFLATED) as zout:
        for name, data in files.items():
            zout.writestr(name, data)
    os.replace(tmp, docx_path)
    print(f'Footnote [{footnote_id}] added: {footnote_text[:60]}...')
```

## 公式（OOXML Math）

### OOXML 数学公式结构

| 元素 | 作用 |
|------|------|
| `m:oMathPara` | 显示公式（独占段落，居中） |
| `m:oMath` | 内联公式（嵌在文本中） |
| `m:r` → `m:rPr` + `m:t` | 公式文本（**m:rPr 可为空但必须存在**） |
| `m:m` | 矩阵 |
| `m:mPr/mcs/mc/mcPr/m:count` | 矩阵列数（m:val 属性） |
| `m:mr` | 矩阵行 |
| `m:e` | 矩阵元素 |
| `m:d` + `m:dPr/begChr/endChr` | 分隔符（括号） |
| `m:f` / `m:num` / `m:den` | 分式（分子/分母） |
| `m:sSup` / `m:sSub` / `m:sSubSup` | 上下标 |

### 提取公式前的检查步骤（必须严格按顺序！）

1. `ls Inputs/*.docx` 确认所有待处理文件及修改时间
2. 对**每个文件**逐段扫描 `m:oMath`，不假设公式在哪个文件
3. 找到后用 `pretty_print` 完整打印 OOXML 结构
4. 将 WPS 原生公式作为模板，比对手工构建的差异

### WPS 兼容规则（手工构建公式时必须遵守）

1. **每个 `m:r` 必须包含 `<m:rPr/>`**（即使为空元素），否则 WPS 不渲染
2. 矩阵列数通过 `m:count m:val="N"` 指定
3. 括号通过 `m:begChr/m:endChr` 的 `m:val` 属性指定字符
4. 命名空间声明在顶层 `m:oMathPara`/`m:oMath` 上，不在子元素上重复
5. 缺少 `m:rPr` 是 WPS 显示空白而 Office Viewer 正常的**根因**

### 项目中的公式工具（build_generated.py 自动生成）

| 函数 | 作用 |
|------|------|
| `formula_text(xml_str)` | 提取公式纯文本，用于生成注释 |
| `formula_remove(xml_str, target)` | 删除包含指定文字的子元素 |
| `formula_replace(xml_str, old, new)` | 替换公式中的文字 |

修改流程：查看公式注释定位 → 调工具函数改 XML → 重跑脚本。不写一次性 fix 脚本。

### LaTeX 公式转换器（latex_omath.py，独立模块）

`latex_to_omath(latex_str)` — 将 LaTeX 数学语法直接转换为 OOXML，像写 `.tex` 文件一样编辑公式。

**用法**：
```python
from latex_omath import latex_to_omath, body_latex

# 在 build_generated.py 中：
body_with_formula("", [latex_to_omath(r"\int_0^\infty e^{-x^2} dx = \frac{\sqrt{\pi}}{2}")])
body_latex(doc, "", r"\begin{pmatrix} a & b \\ c & d \end{pmatrix}")
```

**支持的 LaTeX 构造**：
- 上下标: `x^2`, `x_1`, `x_1^2`
- 分式/二项式: `\frac{a}{b}`, `\binom{n}{k}`
- 根式: `\sqrt{x}`, `\sqrt[n]{x}`
- 求和/积分/乘积: `\sum_{i=1}^n`, `\int_a^b`, `\prod`, `\iint`, `\oint`
- 定界符: `\left( ... \right)`, `\left[`, `\left\{`, `\left|`, `\left\langle`
- 希腊字母: `\alpha`~`\omega`, `\Gamma`~`\Omega`
- 符号: `\infty`, `\partial`, `\nabla`, `\times`, `\pm`, `\cdot`, `\forall`, `\exists`, `\in`, `\to`, `\Rightarrow` 等 60+
- 函数: `\sin`, `\cos`, `\tan`, `\log`, `\ln`, `\exp`, `\det`, `\lim`
- 重音: `\hat`, `\bar`, `\vec`, `\dot`, `\ddot`, `\tilde`, `\widehat`, `\widetilde`
- 上下线: `\overline{abc}`, `\underline{abc}`
- 矩阵: `\begin{pmatrix}`, `\begin{bmatrix}`, `\begin{cases}` 等
- 括号: `\overbrace{expr}^{label}`, `\underbrace{expr}_{label}`
- 框: `\boxed{expr}`
- 文本: `\text{text}`, `\mathrm{...}`, `\mathbf{...}`

**content.json 配置**：在 math 条目中添加 `latex` 字段，流水线自动使用 LaTeX 转换：
```json
{"math": [{"type": "display", "xml": "...", "text": "E=mc^2", "latex": "E=mc^2"}]}
```
优先使用 `latex` 字段，无 `latex` 字段时回退到 OOXML 透传（向后兼容）。

### 文本公式重建（当前流水线自动处理）

有些源 docx 中的公式不是 WPS/Word 原生 `m:oMath`，而是普通段落文本，例如 `带宽 = 数量 × 单台带宽`。当前流水线按下面的职责拆分处理：

1. `formula_semantics.py` 先做语义判别，区分普通数量/单位、行内公式、显示公式和疑似碎裂公式。不要只因为段落里有数字、百分号或单位就转公式。
2. `content_parser.py` 识别疑似公式段落、公式定义行和公式续行，写入 `role="formula"`，并标记 `source="text"` 与是否需要编号；普通正文中的保守行内公式会写成 `role="rich_text"` token 流，DOCX 表格单元格中的 `$...$` 行内公式也会在原单元格位置写成可替换的 `rich_text`，即使同一段还混有 Word 原生 OMML。
3. `script_generator.py` 将文本公式归一化为 LaTeX：单位用 `\mathrm{Mbps}`，中文说明用 `\text{...}`，乘号、上下标、比较符号等转成数学语法。
4. `latex_omath.py` 通过 `latex_to_omath(..., display=True)` 生成 `m:oMathPara`，最终 docx 在 WPS/Word 中可作为原生公式编辑。

验证时不要只看外观：应解压 docx 检查 `word/document.xml` 中存在 `<m:oMathPara>`，并渲染到 Word/WPS 或 PDF 检查没有 `[LaTeX error]`、没有文本公式残留、没有 `M|b|p|s` 这类 run 分隔伪影。

注意：
- 不要把整个公式包成一个 `\text{...}`，否则它会变成不可编辑的普通文本外观。
- 多行对齐公式应优先使用真正的 OMML 结构；如果用矩阵伪装对齐，在 WPS 中可能出现竖向分隔线或奇怪换行。
- 行内公式切分必须保守：候选里混入中文连接词、未闭合括号/花括号、中文标点时，宁可保留正文，也不要生成可能失败的 LaTeX。
- QA 中的 `FORMULA_TEXT_FRAGMENTED` 表示源内容已经碎裂或公式对象混入正文，优先回到原始未排版内容、Word 原生公式或 Markdown LaTeX，而不是让生成器猜。
- 通用修复应沉淀到 `content_parser.py` / `script_generator.py` / `latex_omath.py`，不要只改某次输出目录的 `build_generated.py`。

### Markdown 公式与图片

Markdown 输入由 `md_parser.py` 负责：

1. `![图](media/a.png)`、`![图](media/a.png "图片标题")`、`<img src="media/a.png" alt="图">`、`<img data-src="media/a.png">`、`<img srcset="media/a.png 1x, media/a@2x.png 2x">`、PNG/JPG data URI、`![图][id]` + `[id]: media/a.png`、引用定义下一行 title、`![图]` + `[图]: media/a.png` 这类图片路径先按 `.md` 文件所在目录解析，找到并确认可作为图片读取后复制到本次输出目录的 `figures/`，生成脚本只引用复制后的文件；本地路径支持 `%20` 空格、`<media/a b.png>` 包裹、文件名括号、可选图片 title，以及从 Markdown 工具复制来的 `?query` / `#fragment` 后缀；Markdown 表格单元格里的图片不能被 `_strip_md_formatting()` 静默删掉，应挂到表格项的 `table_cell_items`，使用 `role="image"` / `role="missing_image"` 并标记 `location="markdown_table_cell"`，生成 DOCX 时渲染在对应 Word 表格单元格内；本地找不到或引用式图片没有定义时记录到 `content._meta.missing_images` 并由 QA 上报 `CONTENT_IMAGE_MISSING`；本地文件存在但打不开、损坏、扩展名不匹配、格式不支持、不是 `.png` / `.jpg` / `.jpeg` 稳定格式，或 data URI 损坏/类型不支持/MIME 与真实格式不一致时记录 `reason=unreadable` 并由 QA 上报 `CONTENT_IMAGE_UNREADABLE`，下一步是重新导出普通 PNG/JPG 后更新 Markdown 链接；远程 URL 不自动下载，记录 `reason=remote` 并由 QA 上报 `CONTENT_IMAGE_REMOTE_UNSUPPORTED`，下一步是让用户下载到本地并改成相对路径后重跑；代码块里的 `[id]: path` 示例应保留在 code block 中，不能作为图片引用定义删除。
2. `$...$` 和 `$$...$$` 会被提取为 `math` / `role="formula"` 项，摘要和正文都由 `script_generator.py` 渲染为原生 `m:oMathPara`。
3. 清理分页前空段落时，包含 `w:drawing`、`w:pict` 或 `oMath` 的段落不能当作空段落删除，否则图片或纯公式段落会丢失。
4. 单文件 Markdown 开头的 YAML/front format 说明块只用于格式提取，不能进入正文内容流；即使文件以 UTF-8 BOM 开头，或 `# 格式说明` 标题因编码污染变成不可识别文本，只要后续是明显格式规则并由 `---` 分隔，也应在 `md_parser_modules/content_helpers.py` 中剥离。
5. Windows 保存的 UTF-8 BOM Markdown 可能第一行是 `\ufeff---`、`\ufeff# Title`；也可能使用 Setext 一级题名 `Title` 下一行 `===`。YAML/front matter、格式块和标题识别都必须先剥离 BOM，并把非 CJK 一级题名存为 `title_en`、CJK 一级题名存为 `title_cn`，否则结构 QA 会给普通用户造成不必要的 `TITLE_MISSING` 或正文缺失困惑。Setext `---` 暂按分隔线/二级标题保守处理，不作为论文题名。

## 目录（TOC）

当前流水线默认生成可见的静态目录行：
- `script_generator.py` 收集 1-3 级标题并渲染目录文本。
- 目录行使用右对齐制表位和点引导线，页码不会掉到下一行。
- Windows + Word COM 可用时，流水线会先生成一次文档，再用 Word 分页读取正文标题页码，最后重建文档并写入目录页码。
- Word COM 不可用时仍保留可见目录行，但页码可能需要后续在 Word/WPS 中核对。

旧的 `insert_toc(doc, title)` Word 域方式仍可作为参考，但主流程应使用当前生成器产出的静态目录逻辑；用户侧微调 `build_generated.py` 时也应围绕这套静态目录实现调整。

## 图清单与表清单（LOF/LOT）

引擎在 TOC 之后、正文之前自动生成图清单（List of Figures）和表清单（List of Tables），当内容中存在图/表题注时自动触发。

### 题注收集

`collect_figure_entries()` 和 `collect_table_entries()` 位于 `runtime_toc.py`，从三个来源收集题注：

1. **已配对的图** — `role="figure"` 的 paragraph item，`caption` 字段即题注文本
2. **未配对的题注** — `role="figure_caption"` / `role="table_caption"` 的 paragraph item
3. **Legacy fallback** — section heading 本身是题注且 section 有 `images` 列表（旧解析器路径）

收集时跳过 front matter（摘要/关键词），但包含附录中的图/表。

### 清单行渲染

`add_list_entry(text, page_text, entry_role)` 与 `add_toc_line()` 同模式：
- 右对齐制表位 + 点引导线（`WD_TAB_LEADER.DOTS`）
- 题注文本 + `\t` + 页码
- profile 通过参数传入（`lof_entry` / `lot_entry`），无层级缩进

### 样式 profile

| Role | 默认继承 | 说明 |
|------|---------|------|
| `lof_heading` | `toc_title` | 图清单标题（黑体 16pt 居中） |
| `lot_heading` | `toc_title` | 表清单标题 |
| `lof_entry` | `toc1` → `toc2` → `body` 安全 fallback | 图清单条目 |
| `lot_entry` | 同上 | 表清单条目 |

模板检测支持中文变体（图清单/表清单/图目录/表目录/插图索引/附表索引）和英文变体（List of Figures/List of Tables/Figure Index/Table Index）。

### 页码解析

双轮构建模式（与 TOC 相同）：
1. Pass 1: `build_document({}, {})` — 生成无页码文档
2. `_infer_caption_pages_from_word_com()` — Word COM 打开 Pass 1 文档，扫描所有段落（跳过含 `\t` 的 LOF/LOT 行本身），匹配题注文本并获取 `para.Range.Information(3)`（绝对页码）
3. 归一化：`v - first_heading_page + 1`（与 TOC 的 `_infer_heading_pages_from_word_com()` 使用相同偏移）
4. Pass 2: `build_document(page_map, cap_page_map)` — 重建文档写入页码

**关键陷阱**: Word COM 段落文本的 `\t` 检查必须在 `.strip()` **之前**进行。`.strip()` 会删除尾随 tab，使空页码的 Pass 1 清单行与正文题注无法区分，导致所有题注被误判为在 LOF 页面上，页码全部归一化为 1。

### 分页结构

```
TOC → doc.add_page_break() → 图清单 → doc.add_page_break() → 表清单 → section break (decimal, from 1) → 正文
```

LOF/LOT 页面继续 TOC 的 `upperRoman` 页码。正文通过 `add_section_with_header('decimal', 1)` 切换到阿拉伯数字。

## 排查指南：流水线生成结果不对怎么办

本节记录了历次排查中发现的典型错误根因和修复方法。**每次排查完新问题都应追加到此，防止同类问题再次出现。**

### 排查铁律

1. **先定位生成结果，再改核心引擎** — `build_generated.py` 可以读，但它是中间产物；确认根因后应改 `script_generator.py`、`format_extractor.py`、`content_parser.py` 或 `md_parser.py`
2. **对比三个来源**：模板规定（基础操作.md）→ `格式提取.md`/`内容提取.md` → `build_generated.py` 中的 helper 函数
3. **debug 时直接检查 `format.json` / `content.json` / 生成脚本里的 DATA**，看实际传入引擎的值
4. **注意终端编码** — Windows 终端可能乱码，用 `repr()` 查看实际字符串内容

### 已确认的 Bug 根因（按发现时间排列）

#### Bug 1: `WD_ALIGN_PARAGRAPH.DEFAULT` 不存在
- **现象**：`AttributeError: DEFAULT`
- **根因**：`format_extractor.py` 对无法确定的对齐返回 `'DEFAULT'`，但 `WD_ALIGN_PARAGRAPH.DEFAULT` 在 python-docx 中不存在
- **修复位置**：`script_generator.py` 生成页眉代码时，将 `DEFAULT` 映射为 `CENTER`

#### Bug 2: `sync_tracker.py` 导入失败
- **现象**：`ModuleNotFoundError: No module named 'pipeline'`
- **根因**：生成脚本写的 `from pipeline.sync_tracker import SyncTracker`，但输出目录没有 `pipeline` 包结构
- **修复位置**：`script_generator.py` 将 `sync_tracker.py` 复制到输出目录，import 改为 `from sync_tracker import`

#### Bug 3: 正文采样用错 JSON key（最严重）
- **现象**：body() 函数字体=黑体、字号=16pt、对齐=CENTER（全错）
- **根因**：`script_generator.py` 采样正文时读取了错误的 JSON key：
  ```
  错误: p.get('line_spacing_val')  →  正确: p.get('ls')
  错误: p.get('alignment')         →  正确: p.get('align')
  错误: p.get('first_indent_cm')   →  正确: p.get('indent')
  ```
  格式提取器写入的 key 与脚本生成器读取的 key 不一致，导致 ls/align/indent 永远取到默认值
- **修复位置**：`script_generator.py` 正文采样段，统一使用 `ls`/`align`/`indent`
- **教训**：JSON key 命名变更时，必须同时 grep 所有读写方确认一致

#### Bug 4: 固定行距被当作倍行距
- **现象**：`pf.line_spacing = 1.5` 而非 `Pt(28.0)`
- **根因**：模板规定"行距固定值28磅"，但正文文字说明解析只检测 `1.5倍` / `双倍`，不识别 `固定值XX磅`
- **修复位置**：`script_generator.py` 正文文字说明解析，增加 `固定值\s*(\d+)\s*磅` 正则
- **教训**：中文排版术语多样（固定值/固定行距/最小值），模式匹配要考虑同义表述

#### Bug 5: CJK 字体被 Latin 字体覆盖
- **现象**：中文论文正文字体显示 Times New Roman 而非宋体
- **根因**：OOXML 采样只提取 `w:ascii` 字体（拉丁字体），中文段落运行时 ascii 常为 TNR，CJK 字体（`w:eastAsia`）被忽略
- **修复位置**：`script_generator.py`，检测到 CJK 字体后，如果 body_font 是拉丁字体则优先用 CJK 字体
- **教训**：中英文混排文档有双字体体系（ascii + eastAsia），必须分别处理

#### Bug 6: 标题字体被 body_font 覆盖
- **现象**：heading1/2/3 全部使用 body_font，而非各自的黑体
- **根因**：标题函数生成代码硬编码 `P['body_font']`，忽略了 `hl['font']`（从模板文字说明提取的标题字体）
- **修复位置**：`script_generator.py` 标题生成段，改为 `hl.get('font', P['body_font'])`
- **教训**：标题格式应该有独立的字体来源，不应混用正文参数

#### Bug 7: 缩进检测遗漏同义表述
- **现象**：正文段落无首行缩进
- **根因**：模板写"缩进2个**汉字**符"，代码匹配 `'2字符'`，但文中是 `'2个汉字符'`（多了"个汉"）
- **修复位置**：增加 `'2个汉字' in body_text` 条件

#### Bug 8: 标题文字说明被目录格式污染
- **现象**：一级标题检测为 宋体四号 14pt（应为 黑体三号 16pt）
- **根因**：P74 的目录格式说明（"一级标题 宋体四号加粗"）和 P114 的实际格式说明（"一级标题三号黑体居中加粗"）都匹配了正文筛选条件。P74 更长，胜出。
- **修复位置**：增加 `黑体` 优先匹配（实际标题用黑体，目录描述用宋体）
- **教训**：模板中的"目录格式说明"和"正文格式说明"容易混淆，需要额外的区分逻辑

#### Bug 9: 封面学校编码表丢失
- **现象**：生成的封面没有"学校编码"行
- **根因**：`_parse_text_instructions` 只从 Table 0 提取字体，没有保存表格内容；封面生成时直接跳过 Table 0
- **修复位置**：提取并存储 Table 0 的 label/value；封面生成时在标题表格前插入学校编码表

#### Bug 10: 封面标签字体取错来源
- **现象**：封面标签用 TNR 12pt，而非模板的 楷体 16pt
- **根因**：标签字体从 Table 0（学校编码表，TNR 12pt）取，而非 Table 1（信息表，楷体 16pt）
- **修复位置**：优先从 Table 1 取标签字体

#### Bug 11: 机构名称被误识别为论文标题
- **现象**：论文标题显示为模板封面/授权页中的机构名称（如“某高校学位评定委员会”）
- **根因**：`content_parser.py` 标题检测仅靠字号筛选（≥14pt），不区分语义。学院评委会印章正好是大字号，被误判为论文标题
- **修复位置**：
  - `content_parser.py`：增加机构关键词排除（委员会、学位评定、原创性声明…）
  - `script_generator.py`：封面和正文标题处二次校验，拒绝机构名称
- **教训**：纯格式特征（字号）可能误判，需要结合语义关键词辅助判断

#### Bug 12: 已排版源 DOCX 的目录、占位符和公式碎片被静默放过
- **现象**：客户把上一轮生成的 `最终论文.docx` 再当作内容输入后，QA 一开始没有拦住“源目录残留、`[报名序号]` 占位符、公式编号重复、公式变量碎片”等问题；部分目录页数字或公式碎片还可能被误判为章节标题。
- **根因**：
  - `content_parser.py` 没有识别并跳过源文档中的静态目录块。
  - 标题候选没有排除 `[报名序号]` 这类未填写占位符，也没有优先使用“论文题目/题目”后的真实标题。
  - 源 OMML 公式中已有编号 `(1)`，生成器再编号会形成 `(1)(1)`；部分旧生成结果里的公式碎片已经退化成普通短段落。
  - 源 OMML 中部分 `m:r` 缺少 `m:rPr`，WPS/strict conformance 会报兼容问题。
- **修复位置**：
  - `content_parser.py`：跳过源 TOC，记录 `source_toc_skipped_paragraphs`；记录 `source_placeholders`；剥离源公式尾部编号但保留 numbered 标记；对明显正文式 OMML 做降级。
  - `script_generator.py`：过滤封面占位行；渲染 OMML 前补齐 `m:rPr`；公式编号只输出一次。
  - `qa_checker.py`：新增 `CONTENT_TOC_POLLUTION`、`UNFILLED_PLACEHOLDER_TEXT`、`FORMULA_NUMBER_CONFLICT`、`FORMULA_TEXT_FRAGMENTED`，让坏输入直接阻断流水线。
  - `regression_suite.py`：增加合成回归覆盖源 TOC、占位符、重复编号、公式碎片和 OMML WPS 兼容。
- **教训**：用户可能把“已排版过的输出稿”重新当作内容源。流水线既要尽量修复可自动归一化的结构，也要把需要用户回到源稿处理的问题明确报出来，不能只生成一个看似成功的 DOCX。

#### Bug 13: 极小文字图片被放大成整页宽
- **现象**：最终论文后几页出现“光伏容量”“风电容量”“月产量”等巨大的模糊文字图片，甚至单个汉字占满大半页。
- **根因**：源 DOCX 里已经存在 76×18、53×18、15×15、20×20 这类低分辨率图片对象，它们通常是图表坐标轴文字、图例文字或粘贴对象碎片。旧生成端把所有正文图片统一缩放到版心宽，导致这些碎片被放大成整页宽。
- **修复位置**：
  - `script_generator.py`：`fit_picture_dimensions()` 对极小图片使用原始尺寸，不再放大到版心宽。
  - `qa_checker.py`：新增 `LOW_RES_IMAGE_FRAGMENT`，检测低分辨率正文图片碎片并给出用户修复步骤。
  - `regression_suite.py`：增加 76×18 图片回归，确认 QA 会报错且生成端不会上采样到整页宽。
- **教训**：图片数量匹配不等于图片语义正确。已排版稿中的拆碎图片必须作为输入质量问题暴露给用户，而不是靠缩放策略掩盖。

#### Bug 14: 中文正文含公式符号时被整段公式化
- **现象**：正文段落只要含 `=`、`%`、`MWh`、`u(t)` 等符号，就可能被误判为显示公式；反过来，真正的行内公式如果切分过宽，又会吞入中文连接词或未闭合集合，生成 `[LaTeX error]`。
- **根因**：旧规则主要看字符形态，没有先做“数量文本 / 行内公式 / 显示公式 / 碎裂公式”的语义判别；QA 也会把 `rich_text` 整段当成公式编号冲突来源。
- **修复位置**：
  - `formula_semantics.py`：新增公式语义层，普通数量和单位保留正文，保守切分行内公式，短小高数学密度且残缺的文本才标记为 `formula_problem`；`$...$` 行内分隔符只接受单美元且语义像公式的内容，`$100$` 这类金额/纯数字和 `$$...$$` 显示公式不会被误切成行内公式。
  - `content_parser.py`：普通正文中的行内公式输出为 `rich_text` token 流，疑似碎裂公式进入 QA 问题项而不是强行转公式。
  - `qa_checker.py`：编号冲突和公式碎片检查跳过 `rich_text` 整段，只检查真正公式块或明确的 `formula_problem`。
  - `regression_suite.py`：增加公式语义、行内公式切分、中文连接词/未闭合集合防误切、金额/显示公式分隔符防误切等回归。
- **教训**：中文论文里的数字、单位和符号不等于公式。公式提取要“语义先行、保守切分、失败可回退”，不能让转换器替解析器猜上下文。

#### Bug 15: 修复公式碎片时过度猜测
- **现象**：源目录没有分页且目录项无页码时，部分目录项会泄漏成正文标题；有分页边界的源目录里若混入公式页码碎片（如重复的 `41 .5`），目录残留可能污染正文；真实的“目录”章节又可能被误删；`∑24 + x_i` 这类碎裂求和被硬补成 `t=1`；修复 WPS 竖线时又可能把 `\mathrm{...}` 合并成默认数学斜体。
- **根因**：目录跳过只依赖分页边界或单行形态，缺少对目录块范围、异常目录碎片和真实 Heading 样式目录章节的联合判断；显式 Word Heading 样式没有优先进入标题识别；求和修复没有从表达式推断索引；OMML run 合并只看文本节点，没比较 `m:rPr` 样式。
- **修复位置**：
  - `content_parser.py`：看到“目录”后先前瞻判断目录块；有分页/分节边界且确认是源目录时跳到边界；若“目录”自身是显式 Heading/标题样式，则保留为真实章节；无边界时用目录项序号、重复标题和真实标题退出，避免吞正文；有边界但出现重复公式/页码碎片时继续跳到边界，避免源目录污染正文。
  - `content_parser.py`：显式 Heading/标题样式优先参与标题识别，真实“目录”章节不会因为前置正文过滤而丢失。
  - `content_parser.py`：求和碎片只在能从 `P(t)`、`x_i` 等表达式推断索引时自动修；对 `24 / u(t)=... / t=1`、`Cton = / 1 / Qtarget / 24 / t=1 / ...`、`max / α / 24 / QNH3... / t=1` 等拆行公式做通用重组；无表达式索引时只允许继承近邻 `t=1, upper=24` 的时间求和上下文，无法推断时输出 `formula_problem`。
  - `latex_omath.py`：多个 `m:r` 只有在 `m:rPr` 完全一致时才合并；混合样式保留分组但显式隐藏 `sepChr`。
  - `qa_checker.py`：低分辨率图片只在更像文字/公式碎片时升级为 `LOW_RES_IMAGE_FRAGMENT`，避免二维码、小图标等合法小图误报。
  - `regression_suite.py`：增加无页码无分页目录、带分页真实“目录”章节、源目录重复公式碎片、小图误报、求和/分式/优化目标拆行修复、索引推断/不猜测、OMML 样式保留回归。
- **教训**：自动修复的底线是不能改变数学语义；不能确定时要明确报错，而不是生成一个看起来更漂亮但含义可能错的公式。

#### Bug 16: Markdown 格式说明块泄漏为正文
- **现象**：纯 Markdown 模式中，YAML front matter 后的 `# 格式说明` 若在 Windows/终端编码链路中变成 `# ????`，下一行 `正文：Times New Roman，小四号，1.5倍行距。` 会被当成正文段落；生成端不会渲染这类格式说明，strict conformance 因此报 `CONTENT_PARAGRAPH_MISSING`。
- **根因**：旧 `_skip_format_section()` 只识别中文标题文本，没有识别“开头标题 + 明显格式规则 + `---` 分隔线”的结构；标题被编码污染时，内容解析器无法知道该块仍是格式说明。
- **修复位置**：
  - `md_parser_modules/content_helpers.py`：新增格式标题文本提取、格式规则行判定和格式块结束位置判断；对开头格式块采用结构化剥离，即使标题文本受污染，只要规则行和分隔线明确，也不进入正文。
  - `regression_suite_modules/md_cases.py`：新增 YAML + 污染格式标题 + `---` + 正文的回归，确认 `title_info` 使用真实论文标题，正文段落不包含 `Times New Roman`、`1.5` 或 `---`。
- **教训**：纯 Markdown 的“格式”和“正文”边界要按结构识别，不能只依赖标题文字；格式说明只能服务 `format.json`，绝不能进入 `content.json` 和 strict QA 的期望正文。

#### Bug 17: BOM + YAML 会让格式块误作论文题名
- **现象**：Windows 保存的 Markdown 若以 UTF-8 BOM 开头，且第一段是 YAML front matter，旧逻辑没有把 `\ufeff---` 识别为 front matter 起点，后续编码污染的格式说明标题会被当成论文题名，格式规则也可能进入正文。
- **根因**：`_skip_format_section()` 只在标题检测里剥离 BOM，没有在 YAML delimiter、空行和格式块起点判断前统一剥离 BOM。
- **修复位置**：
  - `md_parser_modules/content_helpers.py`：在 YAML/front matter、格式块起点、格式块结束和前置空行判断中统一使用 `_strip_bom_prefix()`。
  - `regression_suite_modules/md_cases.py`：新增 BOM + YAML + 格式说明块 + 行内/显示公式 + 缺图组合回归，确认题名、正文、公式、图片复制和 `CONTENT_IMAGE_MISSING` 下一步都保持正确；远程 Markdown 图片另有 `CONTENT_IMAGE_REMOTE_UNSUPPORTED` 回归，确认报告提示用户下载到本地并改成相对路径。
- **教训**：BOM 是文件级边界问题，不只是标题问题。先清理文件开头控制字符，再判断 YAML、格式说明和论文正文边界，才能避免小白用户用 Windows 编辑器保存后出现误报或静默漏内容。

#### Bug 18: 内容摘要把结构化图片显示成不明标签
- **现象**：表格单元格图这类 `role="image"` 内容项已经被抽取和渲染，但 `内容提取.md` 只把 `role="figure"` 显示为 `[图片]`，普通结构化图片会落成 `[结构化内容]`，普通用户无法确认图片是否被识别。
- **根因**：`pipeline_runner/artifacts.py` 的 `_paragraph_summary()` 只识别 `figure`，没有覆盖 `role="image"` 或带 `image` / `filename` / `asset` 字段的结构化内容。
- **修复位置**：
  - `pipeline_runner/artifacts.py`：统一把 `figure`、`image` 和带图片字段的结构化项显示为 `[图片] <文件名>`，有 caption 时保留 caption。
  - `regression_suite_modules/pipeline_cases.py`：扩展 `pipeline_artifacts_write_format_and_content_handoffs`，确认表格单元格图摘要不再出现 `[结构化内容]`。
  - `regression_suite_modules/content_cases.py`：扩展表格单元格图 + 页眉/页脚图组合边界，确认正文表格图渲染一次，non-body 图以 `NON_BODY_IMAGE_UNSUPPORTED` 明确阻断。
- **教训**：QA 计数正确还不够，给用户看的摘要也必须用用户能理解的角色标签。否则小白用户会误以为图片没有被处理，或不知道下一步该检查哪里。

#### Bug 19: 扫描 PDF 模板太晚才阻断
- **现象**：扫描件或不可复制文字 PDF 模板已经被识别为 `PDF_TEMPLATE_UNSUPPORTED`，但旧流水线仍继续抽取内容、生成 `build_generated.py`、构建 `最终论文.docx`，最后才在结构 QA 阶段失败。普通用户会看到一个“已经生成”的 DOCX，容易误以为只是小问题。
- **根因**：`run_pipeline.py` 只在最终 QA 阶段运行格式问题检查，缺少“模板格式已不可用时不应继续生成”的早期失败出口。
- **修复位置**：
  - `pipeline_runner/artifacts.py`：新增早期格式阻断交接，复用结构 QA 的 `run_format_checks`、`build_report` 和 `write_reports`，在 `PDF_TEMPLATE_UNSUPPORTED` 时写出 `qa_report.md/json`、`qa_repair_plan.md/json` 和 `qa_fix_prompt.txt`。
  - `run_pipeline.py`：模板画像写出后立即检查格式阻断；若 PDF 模板不可用，写出 `agent_summary.md/json` 并停止在内容解析和脚本生成前。
  - `regression_suite_modules/pipeline_cases.py`：新增完整流水线回归，确认扫描 PDF 模板不会生成 `build_generated.py` 或误导性的最终 DOCX，且 `agent_summary` 保留 DOCX/文字说明 PDF/OCR 下一步。
- **教训**：可提前证明“输入不可用”的边界必须提前阻断。小白用户需要的是下一步，而不是一个后来才被 QA 否决的半成品。

#### Bug 20: 缺 Poppler 时误导用户去 OCR PDF

#### Bug 40: 结构化表题没有跟随横向表格进入同一 section
- **现象**：内容解析器已经把表题识别成 `{"role": "table_caption"}` 时，后续横向宽表会单独进入 landscape section，但表题可能先写在纵向正文 section 中。用户看到的是“表题在前一页或纵向页，表格在后一页横向页”，只能靠肉眼发现。
- **根因**：`runtime_body.py` 只把“字符串表题 + 后续表格”作为同一个横向表格组处理，遗漏了结构化 `table_caption` item + 后续表格的常见路径。
- **修复**：
  - `script_generator_modules/runtime_body.py`：正文渲染循环遇到 `table_caption` 后紧跟普通表格时，先开启表格源 section，再写表题和表格，最后恢复后续正文 section；非横向表格仍保持原有顺序。
  - `regression_suite_modules/generator_cases.py`：新增结构化表题 + 横向表格回归，按 `document.xml` 顺序确认 section 断点在表题前、横向 section 结束在表格后，后续正文回到纵向 section。

#### Bug 41: 相邻横向表被短说明拆成多个 section
- **现象**：连续两个横向宽表中间只有一句短说明时，旧生成逻辑会先结束第一个横向 section，把短说明写回纵向正文，再为第二个宽表重新创建横向 section。用户看到的是横纵页面来回切换、短说明离开表格组，严重时还会出现多余空白页。
- **根因**：`runtime_body.py` 按单个表格启动和结束 `begin_table_source_section()` / `end_table_source_section()`，没有识别“横向表格组”的上下文；短说明虽然语义上属于两个宽表之间的桥接文字，却被普通正文路径渲染。
- **修复**：
  - `script_generator_modules/runtime_body.py`：新增横向表格组识别，只对带 `source_section_page_setup` 的横向表生效；当两个横向表之间只有短普通说明时，在同一个 landscape section 内连续渲染表题、表格和说明，再统一恢复后续正文 section。

- **剩余边界**：长说明块、图片/独立公式/列表夹在横向表之间，以及横向表跨多页时的 Word/WPS 实际分页仍需要 visual QA 或后续分组策略增强。
- **教训**：横向表不是孤立元素，真实论文常把多个宽表和短说明作为一组。引擎要识别这种最小安全组，减少用户看到的异常换页和空白页。

### 修改流水线引擎时的检查清单

当修改 `script_generator.py`、`format_extractor.py`、`content_parser.py` 中的任何一个时：

1. **JSON key 一致性**：grep 所有读写该 key 的代码，确保名称一致
2. **变量定义顺序**：Python 代码生成时，变量必须在其引用前定义（`ls_code` 在 DEFAULT STYLE 前定义）
3. **中文模式匹配**：检查所有同义表述（如 "2字符"="2个汉字"=""两个字符"）
4. **字体来源优先级**：text_rules > OOXML 采样 > 硬编码默认值
5. **终端编码**：Windows 上验证时用 `repr()` 确认中文字符串内容

## PDF 模板适配规则

- `Templates/` 可以放 `.docx` 或 `.pdf` 模板；`.pdf` 仍由 `format_extractor.py` 进入格式提取阶段。
- PDF 模板需要 Poppler 命令行工具：`pdfinfo`、`pdftotext`。视觉 QA 还需要 `pdftoppm`。
- 如果缺少 `pdfinfo` 或 `pdftotext`，必须在生成脚本前进入 QA error `PDF_TEMPLATE_DEPENDENCY_MISSING`，`qa_repair_plan.json` 使用 `resume_scope=environment`，提示先安装/修复 Poppler 并确认命令在 PATH 中可用，然后重跑完整流水线；不要误导用户去 OCR 一个本来可用的文字 PDF。
- 如果 PDF 需要打开密码，或禁止复制/提取文字，必须在生成脚本前进入 QA error `PDF_TEMPLATE_PROTECTED`，`qa_repair_plan.json` 使用 `resume_scope=input_files`，提示用户解除密码/复制权限，或从原始文档重新导出无密码、可复制文字的 PDF，也可以改用 DOCX 模板后重跑。
- 如果 `pdfinfo` 或 `pdftotext` 已运行但读取失败，必须在生成脚本前进入 QA error `PDF_TEMPLATE_READ_FAILED`，提示用户先确认 PDF 能正常打开；损坏或导出不完整时从原始文档重新导出可复制文字 PDF，或改用 DOCX 模板后重跑。
- 文字说明型 PDF：按“正文、标题、字号、行距、页边距”等规则提取为格式说明。
- 稀疏文字说明 PDF：如果缺少标题、图表题注、参考文献等关键规则，必须进入 `PDF_TEMPLATE_INSTRUCTION_INCOMPLETE` warning，`qa_report`、`qa_repair_plan` 和 `agent_summary` 要点名缺失规则，提示用户补充说明、改用更完整 DOCX 模板，或生成后重点人工核对。
- 横向 PDF 模板：如果 PDF 模板首页宽度大于高度，必须进入 `PDF_TEMPLATE_LANDSCAPE_PAGE` warning，提示用户用 Word/WPS 核对最终 DOCX 页面方向、页边距、正文和表格是否被压缩；如果用户实际不想横向页面，应换纵向 DOCX/PDF 模板后重跑。
- 精排样张型 PDF：根据页面尺寸、文字 bbox、字号和对齐方式估计模板画像，必须进入 `PDF_TEMPLATE_VISUAL_APPROXIMATION` warning，`template_profile.md` 标记 `pdf_template_visual_approximation`，并提示用户用 Word/WPS 核对估算版式；同时保留有限置信度提示。
- 扫描件/不可复制文字 PDF：必须在模板画像后、内容解析和 `build_generated.py` 生成前进入 QA error `PDF_TEMPLATE_UNSUPPORTED`，提示用户提供 DOCX 模板、文字说明 PDF，或先 OCR 后重跑，不能生成一个看似可交付的默认格式 DOCX。
- PDF 不能提供完整 Word 样式树、页眉页脚/封面 OOXML 结构；最终交付仍要用 Word/WPS 或 `--qa-level visual` 做视觉核对。
